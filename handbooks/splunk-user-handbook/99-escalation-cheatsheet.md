# Chapter 99 ‚Äî When to escalate to an admin: consolidated cheatsheet

> Every chapter of this handbook ends with a short "When to escalate to an admin" section. This page consolidates all of them into one printable reference. When the answer to "should I file a ticket or rewrite my search" is not obvious, scan the table, find the observable symptom that matches yours, and copy the corresponding ask into your ticket. The chapter columns are there so you can jump back to the relevant chapter for the context the admin will want.

## Quick refresher

Nothing chapter-specific here ‚Äî see Foundations.

## Major good practices

1. **Frame every request as `observable symptom ‚Üí ask ‚Üí justification`.** "I think indexing is broken" is a hypothesis; "`| stats count by user` on `sourcetype=linux_secure` returns 0 events with `user` populated over the last 24h" is a symptom. Admins triage by symptom, not by hypothesis. State what you observed, what you want them to do, and why; in that order. The "What to ask for" column of the table is written in that shape so you can lift it verbatim.
2. **Cite the chapter you came from.** A line like "Chapter 4 ‚Äî `tstats` against an unaccelerated data model ‚Äî see [`04-spl-acceleration-tstats-datamodels.md`](04-spl-acceleration-tstats-datamodels.md)" tells the admin which mental model you are operating in. It also lets them push back with a chapter reference of their own if your reading is off. The cheatsheet's `Source chapter` column exists precisely so you can paste this into a ticket without thinking.
3. **Bring a Job Inspector link or an SPL snippet whenever applicable.** The Job Inspector view captures the search head, the dispatch ID, the timing breakdown, the `normalizedSearch`, and the event/result counts. One link saves three round-trips. If the issue is configuration rather than search behavior (a missing field, a wrong sharing scope), bring the smallest SPL that reproduces the symptom ‚Äî three lines beat a paragraph.

## Anti-patterns to ban

1. **"Splunk is slow" with no scope, no time range, no example search.** This phrase has triggered more wasted admin hours than any other. The admin cannot reproduce, cannot measure, cannot prioritize. If you cannot point at a job, you cannot escalate; instead, run one bad search, capture the Job Inspector link, and escalate that.
2. **Asking for the `admin` role "just for now".** The right grain of escalation is a capability or an index, not a role. "Add `schedule_search` to role `analyst`" is a discussable ticket; "make me admin for the afternoon" is not. If you find yourself reaching for the second sentence, the answer is almost always that you want a capability bump, a one-shot done by the admin on your behalf, or a rewrite that does not need admin at all.
3. **Skipping the conversation when the answer is a rewrite, not a capability bump.** The most common failure mode is filing a ticket that asks the admin to raise a limit when a rewrite would have brought you under it. Read the chapter the symptom came from before you file; most chapters explicitly tell you to try the rewrite first. The admin will tell you to do that anyway, plus a day of latency.
